# Overchat · Дневной монитор воронки (автообновляемый, с выбором дат)

Живая страница с ключевой воронкой по суткам. Период выбирается прямо на странице
(даты «с/по» + пресеты 7д/14д/30д/Всё) — графики, таблица и сетка «Покупки и апгрейды»
пересчитываются под выбранный диапазон мгновенно, без запросов в BigQuery.

Данные обновляет **Google Apps Script** раз в день — инкрементально копит историю:

```
Apps Script (триггер ~09:00, ежедневно)
  → BigQuery: пересчёт последних 14 зрелых дней (буфер 2 дня — GA4 лагает)
    → подшивка к накопленной истории (с 2026-06-21)
      → коммит dashboard/data/*.json в репо
        → dashboard/index.html читает JSON, фильтр дат — на клиенте
```

Первый ручной запуск делает полный бэкфилл с `HISTORY_START` (один большой запрос),
дальше ежедневно сканируются только последние ~16 дней — дёшево.

## Что где лежит

| Файл | Что это |
|---|---|
| `index.html` | Страница-отчёт с фильтром дат |
| `data/funnel_daily.json` | Воронка по дням, вся история — **пишет Apps Script** |
| `data/purchases_events.json` | События покупок по юзерам (анонимные хеши + unix-время) — из них сетка метрик считается под ЛЮБОЙ диапазон — **пишет Apps Script** |
| `data/annotations.json` | Полосы релизов, подсветка провалов, баннер — **правится руками** |
| `apps-script/Code.gs` | Скрипт автообновления (вставить в script.new) |
| `apps-script/appsscript.json` | Манифест (таймзона, BigQuery-сервис, скоупы) |

## Установка (один раз, ~10 минут)

1. **Создать скрипт**: [script.new](https://script.new) → вставить содержимое `apps-script/Code.gs`.
   В настройках проекта включить «Show appsscript.json» и заменить его содержимым `apps-script/appsscript.json`
   (поправь `timeZone` на свою — от неё считается час запуска триггера).
2. **Подключить BigQuery**: слева «Services» → «+» → BigQuery API → Add.
   (Скрипт должен жить на Google-аккаунте с доступом к проекту `zinc-hour-447409-k5`.)
3. **GitHub-токен**: GitHub → Settings → Developer settings → Fine-grained tokens → Generate:
   - Repository access: **только этот репозиторий**;
   - Permissions → Contents: **Read and write**. Больше ничего.
   В Apps Script: Project Settings → Script Properties → добавить `GITHUB_TOKEN` = токен.
4. **Проверить ветку**: в `Code.gs` → `CFG.GH_BRANCH` — куда коммитить данные (сейчас `master`,
   дефолтная ветка этого репо). Если переедешь на другую — поправить.
5. **Первый прогон руками**: выбрать функцию `runDaily` → Run → выдать разрешения.
   Это бэкфилл всей истории — может идти пару минут. Проверить: в репо коммиты
   `data: funnel daily refresh ...`, в `funnel_daily.json` заполнились `purchase_onetime`/`purchase_sub`,
   в `purchases_events.json` появился массив `users`.
6. **Включить расписание**: выбрать функцию `setupTrigger` → Run. Всё — дальше само, каждый день ~09:00.

## Как открыть страницу

- **GitHub Pages** (рекомендую): Settings → Pages → Deploy from branch → выбрать ветку и `/ (root)`.
  Страница будет на `https://<owner>.github.io/<repo>/dashboard/`.
  ⚠️ На бесплатном плане Pages для **приватного** репо недоступен (страница публична по ссылке).
  Если палить данные нельзя — открывать локально (ниже) или спросить Claude про альтернативы.
- **Локально**: `cd dashboard && python3 -m http.server 8000` → открыть `http://localhost:8000`.
  Двойным кликом по `index.html` не выйдет — браузер блокирует `fetch` с `file://`.

## Методология (коротко)

- Когорта = день **первого визита лендинга**; каждый следующий шаг строго **после** предыдущего по времени у того же юзера.
- Одноразка = `purchase_onetime` (все методы: карта+Apple+Google). Подписка = `subscription_started`.
- Тест-юзеры исключены (список в `Code.gs` — новые внутренние аккаунты дописывать там), stage-трафик выкинут.
- История копится с `HISTORY_START`; каждый прогон пересчитывает последние 14 зрелых дней
  (даёт нижним шагам дозреть), более старые дни заморожены.
- Сетка «Покупки и апгрейды» считается на клиенте из `purchases_events.json` по выбранному диапазону;
  сравнение — с предыдущим периодом той же длины. «Повтор» и «Onetime→подписка» — floor
  (у покупателей конца диапазона апгрейд ещё не успел случиться).
- ⚠️ Оговорка инкрементальности: юзер, вернувшийся на лендинг спустя >14 дней после первого визита,
  может попасть в две дневные когорты (старую и новую) — GA4 pseudo-id и так живёт недолго, эффект малый.
- Money-truth по суммам — Stripe; GA4 — поведение и конверсии.
- В `purchases_events.json` нет PII: юзеры — усечённые MD5-хеши `user_pseudo_id` (сам по себе анонимный id GA4).

## Аннотации (полосы релизов на графиках)

`data/annotations.json` — руками, прямо в гите:

```json
{
  "banner": {"title": "⚠️ Заголовок", "html": "Текст с <b>жирным</b>"},
  "events": [{"d": "2026-07-17", "color": "teal", "label": "3 оффера"}],
  "bands":  [{"from": "2026-07-09", "to": "2026-07-16", "label": "провал регистрации"}]
}
```

Цвета: `amber` (плохое/регресс), `teal` (релиз/хорошее), `gray` (нейтральное), `violet`.
События вне выбранного диапазона не рисуются. `banner` можно удалить — блок пропадёт.
